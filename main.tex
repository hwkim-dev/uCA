\chapter{ABOUT THIS MANUAL}

\section{Scope}

This manual is the authoritative reference for the
\textbf{\color{oceanblue}pccx v002 Instruction Set Architecture}, the user-facing
binary contract between the host driver running on the Zynq UltraScale+ PS and
the NPU execution complex that lives on the PL.

The ISA is a \textbf{fixed 64-bit} encoding, VLIW in spirit: one instruction
dispatches thousands of MAC cycles across the systolic array. Five opcodes are
defined in v002: GEMV, GEMM, MEMCPY, MEMSET, and CVO. The remaining eleven
opcode slots (\texttt{4'h5}--\texttt{4'hF}) are reserved.

\section{Audience}

This document assumes familiarity with:

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=0.5ex]
    \item Systolic arrays (Weight Stationary / Weight Streaming dataflow).
    \item AXI4 / AXI4-Lite, in particular the Zynq UltraScale+ HP and ACP ports.
    \item Transformer LLM decoding (Q/K/V projection, attention, FFN, RoPE).
    \item Block-floating-point (BFloat16) numerics and INT4 weight quantization.
\end{itemize}

It does \textit{not} attempt to be a tutorial. For onboarding material, start at
the pccx documentation portal: \url{https://hwkim-dev.github.io/pccx/}.

\section{Relationship to the HTML Documentation}

The web portal at \url{https://hwkim-dev.github.io/pccx/} contains the same
ISA tables that this PDF carries, plus live diagrams rendered via Mermaid /
WaveDrom. This guidebook is the \textbf{offline, downloadable, signoff-ready}
version — identical numbers, identical opcodes, typeset for print.

Both sources are regenerated from the authoritative RTL definitions in the
external RTL repository
(\url{https://github.com/hwkim-dev/pccx-FPGA-NPU-LLM-kv260}) at file
\texttt{hw/rtl/NPU\_Controller/NPU\_Control\_Unit/ISA\_PACKAGE/isa\_pkg.sv}.
If this manual ever disagrees with \texttt{isa\_pkg.sv}, the SystemVerilog
package wins and this manual is wrong.

\section{Conventions}

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=0.5ex]
    \item Bit ranges are written \textbf{\texttt{[MSB:LSB]}}, inclusive at both
          ends. \texttt{[63:60]} is a 4-bit field spanning four bits.
    \item Opcodes are always 4 bits wide; values are written in SystemVerilog
          literal form, e.g.\ \texttt{4'h0}.
    \item Pointer / address widths are notated as \texttt{Nb} (e.g.\ 17b = 17
          bits).
    \item \texttt{dest} always precedes \texttt{src} in field ordering, mirroring
          Intel syntax (\texttt{dest = f(src1, src2, \ldots)}).
    \item Reserved bits MUST be written as zero by software. The decoder does
          not check this, but future revisions may assign the fields.
\end{itemize}

\section{Top-level 64-bit Layout}

Every pccx v002 instruction is exactly \textbf{64 bits wide}. The four
most-significant bits are the opcode; the remaining 60 bits are the
per-opcode instruction body.

\section{Opcode Table}

Of the sixteen possible opcodes, five are in use today. The remainder are
reserved for future ISA extensions — decoder behavior on reserved opcodes
is \textbf{undefined} and should not be relied upon.

\begin{longtable}{|c|l|l|p{7.5cm}|}
\hline
\textbf{\color{oceanblue}Opcode} &
\textbf{\color{oceanblue}Mnemonic} &
\textbf{\color{oceanblue}Function} &
\textbf{\color{oceanblue}Primary fields} \\
\hline
\endhead

\texttt{4'h0} & \textbf{GEMV}   & Matrix $\times$ vector                 & dest\_reg, src\_addr, flags, shape\_ptr, size\_ptr, parallel\_lane \\
\hline
\texttt{4'h1} & \textbf{GEMM}   & Matrix $\times$ matrix                  & Identical layout to GEMV (dest\_reg, src\_addr, flags, shape\_ptr, size\_ptr, parallel\_lane) \\
\hline
\texttt{4'h2} & \textbf{MEMCPY} & Device-to-device data movement          & from\_device, to\_device, dest\_addr, src\_addr, aux\_addr, shape\_ptr, async \\
\hline
\texttt{4'h3} & \textbf{MEMSET} & Constant Cache write                    & dest\_cache, dest\_addr, a/b/c\_value \\
\hline
\texttt{4'h4} & \textbf{CVO}    & Complex Vector Op (SFU)                 & cvo\_func, src\_addr, dst\_addr, length, flags, async \\
\hline
\texttt{4'h5}--\texttt{4'hF} & \textit{reserved} & ---                    & Reserved for future extensions. Decoder behavior undefined. \\
\hline
\end{longtable}

\section{μop Decomposition}

Inside the dispatcher, each opcode decomposes into a narrower μop that
carries only the back-end-relevant fields.

\begin{longtable}{|l|p{10.5cm}|}
\hline
\textbf{\color{oceanblue}μop type} & \textbf{\color{oceanblue}Fields} \\
\hline
\endhead
\texttt{gemm\_control\_uop\_t}          & flags (6b) + size\_ptr\_addr (6b) + parallel\_lane (5b) \\
\hline
\texttt{GEMV\_control\_uop\_t}          & Same layout as \texttt{gemm\_control\_uop\_t}. \\
\hline
\texttt{memory\_control\_uop\_t} (49b) & data\_route (8b) + dest\_addr + src\_addr + shape\_ptr + async \\
\hline
\texttt{memory\_set\_uop\_t}           & dest\_cache (2b) + dest\_addr + a/b/c\_value (16b $\times$ 3) \\
\hline
\texttt{cvo\_control\_uop\_t}          & cvo\_func (4b) + src\_addr + dst\_addr + length + flags + async \\
\hline
\end{longtable}

header (opcode + control flags) but differ in payload layout.

\chapter{PER-INSTRUCTION REFERENCE}

This chapter is the normative reference for every v002 opcode. Each section
gives (1) the exact bit-by-bit body layout, (2) the semantics of every flag,
and (3) the typical use case in Transformer decoding.

% -----------------------------------------------------------
\section{GEMV / GEMM (opcode = \texttt{4'h0} / \texttt{4'h1})}

GEMV and GEMM share an identical body layout; only the back-end resources
they target differ (GEMV routes to the 4-wide GEMV lane, GEMM routes to the
32$\times$32 systolic array).

\subsection{Body Layout}

\begin{longtable}{|c|c|l|p{8cm}|}
\hline
\textbf{\color{oceanblue}Bits} &
\textbf{\color{oceanblue}Width} &
\textbf{\color{oceanblue}Field} &
\textbf{\color{oceanblue}Description} \\
\hline
\endhead
\texttt{[59:43]} & 17 & \texttt{dest\_reg}       & L2 address where the result is written. \\
\hline
\texttt{[42:26]} & 17 & \texttt{src\_addr}       & L2 address of the activation source. \\
\hline
\texttt{[25:20]} &  6 & \texttt{flags}           & See \S\ref{sec:gemm-flags}. \\
\hline
\texttt{[19:14]} &  6 & \texttt{size\_ptr\_addr} & Constant Cache index for the size entry. \\
\hline
\texttt{[13:8]}  &  6 & \texttt{shape\_ptr\_addr}& Constant Cache index for the shape entry. \\
\hline
\texttt{[7:3]}   &  5 & \texttt{parallel\_lane}  & Number of core lanes to activate; 0 = all. \\
\hline
\texttt{[2:0]}   &  3 & reserved                 & Must be zero. \\
\hline
\end{longtable}

\chapter{DATA FLOW ARCHITECTURE}

This chapter describes the memory hierarchy and data flow pipeline for the
pccx NPU, from the external DDR4 memory down to the execution units. The
architecture is designed to sustain high bandwidth while crossing multiple
clock domains between the PS and the PL.

\section{Top-level Pipeline}

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=1ex]
    \item \textbf{\color{oceanblue}DDR4 to PS-PL Interface (HP Ports):} \\
    Data is fetched from external DDR4 through four High-Performance (HP)
    ports (HP0--HP3). Each HP port is 128 bits wide and clocked at
    \textbf{250 MHz}.

    \item \textbf{\color{oceanblue}Clock Domain Crossing (Async FIFO):} \\
    The main FPGA compute core is driven by a \textbf{400 MHz} clock. Data
    crossing the PS$\to$PL boundary is safely transferred via Xilinx
    Parameterized Macro (XPM) asynchronous FIFOs configured in
    \texttt{independent\_clock} mode.

    \item \textbf{\color{oceanblue}L2 and L1 Cache Routing:} \\
    FIFO-buffered data is routed to the L2 cache, then loaded into per-core
    L1 caches for low-latency operand access by the execution engines.

    \item \textbf{\color{oceanblue}Execution Engine:} \\
    Systolic arrays (GEMM) and 32-lane MAC banks (GEMV) fetch pre-aligned
    operands directly from L1 cache. Post-process stages (scale, bias,
    \texttt{e\_max} update) write results back to L2.
\end{itemize}

\section{Per-Instruction Dataflow}

\subsection{GEMM}

\begin{enumerate}[label=\textbf{\color{oceanblue}\arabic*.}, leftmargin=*, itemsep=0.5ex]
    \item Dispatcher reads \texttt{shape\_ptr\_addr} / \texttt{size\_ptr\_addr}
          from the Constant Cache to obtain the tile parameters.
    \item Weight Buffer prefetches one tile of weights via the HP ports.
    \item L2 cache \texttt{src\_addr} streams activations into the systolic
          array.
    \item The array runs \textbf{Weight Stationary} $\to$ Accumulator
          $\to$ Post-Process.
    \item Result is written back to L2 cache at \texttt{dest\_reg}.
\end{enumerate}

\subsection{GEMV}

\begin{enumerate}[label=\textbf{\color{oceanblue}\arabic*.}, leftmargin=*, itemsep=0.5ex]
    \item Dispatcher resolves pointers $\to$ per-core shape distribution.
    \item Weight Buffer $\to$ 4 GEMV cores, streaming \textbf{by row partition}.
    \item L2 cache $\to$ per-core L1 cache for activation preload.
    \item 32-lane LUT-based MAC inside each core + 5-stage reduction tree
          $\to$ scalar result.
    \item Post-Process (scale, bias) $\to$ L2 cache or direct SFU FIFO.
\end{enumerate}

\subsection{MEMCPY}

The \texttt{from\_device} / \texttt{to\_device} combination selects one of
the routing paths tabulated in \S3.1. When \texttt{async = 1}, execution
continues to the next instruction immediately; the Global Scheduler tracks
the completion fence.

\subsection{MEMSET}

MEMSET writes directly from the Dispatcher into the Constant Cache — the
only instruction that bypasses L2. A single instruction can write up to
three 16-bit values \texttt{(a, b, c)} at once.

\subsection{CVO}

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=0.5ex]
    \item Dispatcher emits a \texttt{cvo\_control\_uop\_t} to the SFU (single
          BF16 scalar pipeline).
    \item Input vector streams in from L2 cache via \texttt{src\_addr}.
    \item The function pipeline (CORDIC / LUT) produces one result per
          cycle in steady state.
    \item Output goes to L2 cache at \texttt{dst\_addr}, or to the GEMV
          direct FIFO if the next instruction can fuse.
    \item \textbf{Fast Path:} if the SFU consumes the output of the
          preceding GEMV immediately, \texttt{src\_addr} carries a
          \textit{special tag} and the L2 round trip is skipped.
\end{itemize}

\section{Dependencies and Completion}

Inter-instruction dependencies are resolved in the Global Scheduler via
two checks:

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=0.5ex]
    \item \textbf{\color{oceanblue}Address hazard:} read-after-write on
          dest / src L2 addresses.
    \item \textbf{\color{oceanblue}Resource hazard:} occupancy of the
          GEMM / GEMV / SFU resources.
\end{itemize}

Completion of an asynchronous instruction (\texttt{async = 1}) is collected
by the \texttt{fsmout\_npu\_stat\_collector} block and reported to the host
through the AXI-Lite \texttt{STAT\_OUT} register.

\chapter{MEMORY SUBSYSTEM \& CACHE ARCHITECTURE}

To sustain 400 MHz operation and maximize DSP48E2 utilization, the pccx
NPU employs a hybrid memory architecture: deeply buffered asynchronous
FIFOs for deterministic weight streaming, combined with a
Software-Managed Cache (SMC) for dynamic Feature Map (FMAP) and KV token
routing.

\section{Weight Data Streaming via URAM FIFOs}

Unlike traditional address-mapped caches, weight parameters are
continuously buffered through UltraRAM-based asynchronous FIFOs. This
stream-centric architecture gives several advantages:

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=1ex]
    \item \textbf{\color{oceanblue}Automatic CDC (Clock Domain Crossing):}
    By instantiating XPM macros in \texttt{independent\_clock} mode, the
    transition between the 250 MHz AXI domain and the 400 MHz compute
    core is natively and safely resolved without manual synchronization.

    \item \textbf{\color{oceanblue}Control Simplicity:} The NPU dispatcher
    is freed from complex address generation. Execution units fetch
    weights through standard \texttt{valid}/\texttt{ready} handshaking,
    reducing critical-path delays.

    \item \textbf{\color{oceanblue}Massive Stream Depth:} Cascading two
    URAM blocks gives 4096 depth $\times$ 128-bit width per port, or
    about 64 KB. Aggregating all 4 ports provides a 256 KB weight buffer.
\end{itemize}

\section{Physical URAM Allocation and Capacity Maximization}

The Kria KV260 provides 64 URAM blocks (roughly 2.25 MB). The pccx NPU
allocates all available blocks to maximize on-chip bandwidth without
causing synthesis or routing congestion.

\begin{itemize}[label={\color{oceanblue}\textbullet}, leftmargin=*, itemsep=1ex]
    \item \textbf{\color{oceanblue}Weight FIFO Allocation (8 URAMs):} \\
    The 4-port Weight FIFO requires 2 URAM blocks per port for a 128-bit
    datapath.

    \item \textbf{\color{oceanblue}FMAP \& KV Cache Allocation (56 URAMs):} \\
    The remaining 56 URAM blocks are dedicated to the unified FMAP and
    KV Cache. Pairing the 56 blocks for 128-bit data width yields 28
    structural pairs.
    \begin{itemize}[label={\color{graytext}\textendash}, leftmargin=*, itemsep=0.5ex, topsep=0.5ex]
        \item \textbf{Maximum Depth:} 28 pairs $\times$ 4096 = \textbf{114,688}
        \item \textbf{Total Capacity:} 114,688 $\times$ 16 B = \textbf{$\sim$1.75 MB}
    \end{itemize}

    \item \textbf{\color{oceanblue}Architectural Implications:} \\
    The ultra-large unified L2 cache easily accommodates the FMAP
    dimension expansion during large-scale Transformer inference, and
    retains tens of thousands of KV Cache tokens on-chip, eliminating
    aggressive DDR4 swapping.
\end{itemize}

\section{Mathematical Principles of Execution}

The NPU applies integer block-floating-point quantization so that the DSP48E2
blocks can compute exact dot products without any floating-point multipliers.

For inputs aligned to a common Block Floating Point (BF16) scale factor:

$$A = a \times S_{\mathrm{bf16}}, \quad B = b \times S_{\mathrm{bf16}}$$

And INT4 quantized weights:

$$C = c \times S_{\mathrm{int4}}, \quad D = d \times S_{\mathrm{int4}}$$

The original mathematical operation is:

$$\mathrm{Result} = (A \times C) + (B \times D)$$

Applying the scale factors:

$$\mathrm{Result} = (a \times S_{\mathrm{bf16}} \times c \times S_{\mathrm{int4}}) + (b \times S_{\mathrm{bf16}} \times d \times S_{\mathrm{int4}})$$

Factoring out the common scale factors gives the final hardware-optimized
equation:

$$\mathrm{Result} = \underbrace{(a \times c + b \times d)}_{\text{DSP48E2 integer calc}} \times (S_{\mathrm{fmap}} \times S_{\mathrm{weight}})$$

\vspace{0.5em}

Linear memory access calculation for tile walks:

$$\mathrm{Current\_Addr} = \mathrm{Base\_Addr} + (\mathrm{Current\_Row} \times \mathrm{Stride}) + \mathrm{Current\_Col}$$

\chapter{PROGRAMMING EXAMPLES}

This chapter walks through short, real instruction sequences that the host
driver would emit. Each example shows the instruction stream, the
preconditions on Constant Cache state, and the expected side effects on L2.

\section{Example 1 — Single GEMM tile (Q projection for one prompt tile)}

\textbf{Goal:} compute $Q = X W_Q$ for a tile of shape $(M, N, K) = (64, 256, 256)$.

\begin{lstlisting}
# 1. Preload shape + size pointers
MEMSET fmap_shape,   idx=0, a=64,  b=256, c=256   # (M, N, K)
MEMSET weight_shape, idx=0, a=scale_wq, b=zp_wq, c=0

# 2. Load weights if not already streaming
MEMCPY from_device=1, to_device=0, dest=L2_W_BASE,
       src=HOST_WQ_OFFSET, aux=HOST_BASE,
       shape_ptr=0, async=0

# 3. Dispatch the GEMM
GEMM   dest_reg=L2_Q_BASE, src_addr=L2_X_BASE,
       flags={findemax=0, accm=0, w_scale=1},
       size_ptr=0, shape_ptr=0, lane=0        # lane=0 → use all lanes
\end{lstlisting}

\section{Example 2 — Autoregressive step (GEMV + Softmax + RoPE)}

\textbf{Goal:} run one decoder step for a single new token. Uses GEMV for every
projection, CVO for softmax and RoPE.

\begin{lstlisting}
# --- Q, K, V projection (all three use the same X) ---
GEMV dest=L2_Q, src=L2_X, flags=w_scale, size_ptr=0, shape_ptr=0
GEMV dest=L2_K, src=L2_X, flags=w_scale, size_ptr=1, shape_ptr=1
GEMV dest=L2_V, src=L2_X, flags=w_scale, size_ptr=2, shape_ptr=2

# --- RoPE rotation on Q and K ---
CVO  func=CVO_COS, src=L2_Q, dst=L2_Q_R, length=D_HEAD, flags=0
CVO  func=CVO_SIN, src=L2_K, dst=L2_K_R, length=D_HEAD, flags=0

# --- Attention score + softmax ---
GEMV dest=L2_SCORES, src=L2_Q_R, flags={findemax=1, w_scale=0},
     size_ptr=3, shape_ptr=3             # Q · K^T
CVO  func=CVO_EXP, src=L2_SCORES, dst=L2_SCORES,
     length=SEQ_LEN, flags=sub_emax
CVO  func=CVO_REDUCE_SUM, src=L2_SCORES, dst=L2_DENOM,
     length=SEQ_LEN, flags=0
CVO  func=CVO_SCALE, src=L2_SCORES, dst=L2_SCORES,
     length=SEQ_LEN, flags=recip_scale   # divide by denom

# --- Attention-weighted V gather ---
GEMV dest=L2_CTX, src=L2_SCORES, flags=w_scale,
     size_ptr=4, shape_ptr=4             # V · P
\end{lstlisting}

\section{Example 3 — Weight hot-swap for a different layer}

\textbf{Goal:} while the compute path is processing layer $N$, preload the
weights for layer $N+1$ asynchronously so the next layer starts at zero cost.

\begin{lstlisting}
# Fire and forget — continues to next instruction immediately
MEMCPY from_device=1, to_device=0,
       dest=L2_W_NEXT_BASE,
       src=HOST_W_LAYER_NP1_OFFSET,
       aux=HOST_BASE,
       shape_ptr=5, async=1

# ... overlap with layer N GEMMs ...
GEMM   dest=..., src=..., ...
GEMM   dest=..., src=..., ...

# Before starting layer N+1, the scheduler already knows the MEMCPY
# fence has been retired, so no explicit barrier is required.
\end{lstlisting}
